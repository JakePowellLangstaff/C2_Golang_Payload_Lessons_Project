Setting up Go and compiling to .exe

Before writing implant code you need to understand why Go is the right tool for this and how to turn source code into a working Windows binary. This covers the full path from install to a cross-compiled .exe.

Why Go?

Most early offensive tooling was written in C. Python and PowerShell became popular for post-exploitation because they are already on Windows. Go sits between the two: it compiles to native machine code like C, is much faster to write, and produces a completely self-contained binary with zero dependencies on the target machine.

When you compile a Go program, everything needed to run it is baked into the single output file. No runtime to install, no DLLs to carry alongside it, nothing. Drop the .exe on any 64-bit Windows machine and it runs immediately. That is the core reason Go appears regularly in offensive tooling research.

Installing the Go toolchain on Windows

Download the Windows installer from go.dev/dl. Run the .msi and follow the prompts — it sets PATH automatically. Open a new terminal and verify:

Command Prompt or PowerShellshell
go version
# go version go1.22.0 windows/amd64

In VS Code, install the Go extension by Google from the Extensions panel (Ctrl+Shift+X). The first time you open a .go file it will prompt you to install gopls and other tools — accept all. This gives you autocomplete, inline error checking, and auto-formatting on save.

GOPATH location. By default Go stores downloaded modules at C:\Users\YourName\go. You do not need to change this. Run go env GOPATH in a terminal to confirm your exact path.

Your first Go program

Create a new file called hello.go. This is the minimum valid Go program:

hello.goGo
package main

import "fmt"

func main() {
    fmt.Println("Hello from Go")
}

Every Go program starts with package main, which tells the compiler this is a standalone executable rather than a library. The main() function is the entry point. No class, no public static void, nothing else required.

terminalshell
# Run without building a binary first
go run hello.go

# Compile to an executable
go build -o hello.exe hello.go
hello.exe
# Hello from Go

Cross-compilation

Two environment variables control the compilation target. Since you are on Windows, you set them inline before the build command:

PowerShellshell
# Build a 64-bit Windows .exe (already the default on Windows)
$env:GOOS="windows"; $env:GOARCH="amd64"; go build -o payload.exe main.go

# Build a Linux binary from Windows (useful for C2 server if on Linux)
$env:GOOS="linux"; $env:GOARCH="amd64"; go build -o payload main.go

# Reset after cross-compiling
Remove-Item Env:GOOS; Remove-Item Env:GOARCH
GOOSGOARCHOutputWhen to use
windowsamd64.exe (64-bit)All payload builds for modern Windows targets
windows386.exe (32-bit)Older systems or shellcode loaders
linuxamd64ELF binaryLinux lab servers or C2 infrastructure
No MinGW or Windows SDK required. Unlike C cross-compilation, Go handles all target formats natively. The compiler knows how to produce a Windows PE binary from any host OS without external toolchains.

Static binaries and the strip flags

A Go binary compiled without flags includes full debug symbols and DWARF debugging tables. Running Ghidra or strings.exe on the binary reveals function names, variable names, and source file paths. The strip flags remove this information at compile time without affecting runtime behaviour.

PowerShell — standard vs stripped buildshell
# Standard build — includes debug info, ~1.8MB
go build -o payload.exe main.go

# Stripped build — no debug symbols, no DWARF tables, ~1.2MB
# -s removes the symbol table  -w removes DWARF debugging info
go build -ldflags="-s -w" -o payload_stripped.exe main.go
Use -ldflags="-s -w" for all subsequent lessons. From Lesson 02 onward, always use the stripped build. The obfuscation phase demonstrates the before/after difference in Ghidra and with strings.exe.
