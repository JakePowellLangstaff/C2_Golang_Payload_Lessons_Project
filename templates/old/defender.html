How this payload is detected and analysed

Every offensive technique in this project mapped to the defensive tool that catches it — from Sysmon events through static binary analysis to network timing detection.

Why this page exists

Building an offensive tool without understanding how it is detected misses the point of the exercise. This page documents every detection method for every technique in the project. Each phase of the payload is matched to the specific tool, event, or signature that would surface it to a SOC analyst or incident responder.

The analysis covers four layers: host-based detection (Sysmon, Autoruns), network-based detection (Wireshark, Zeek), static binary analysis (PEStudio, FLOSS, Ghidra), and the obfuscation bypass methods that an analyst uses to defeat the techniques implemented in Phase 04. VirusTotal and the reporting process are covered at the end.

Lab setup for this analysis. Sysmon with the SwiftOnSecurity config on the target VM. Wireshark on the operator machine. PEStudio, FLOSS, and Ghidra on an analysis machine.

Static analysis toolkit

These are the tools an analyst uses to examine the payload binary before executing it. Static analysis works on the file on disk without running anything.

PEStudio
PE analyzer — free Windows tool
Examines PE headers, imports, strings, entropy, and indicators in one view. First stop for any Windows binary. Flags suspicious imports (CreateProcess, WriteFile, RegSetValueEx) and calculates section entropy to detect packing.
FLOSS
Obfuscated string solver — Mandiant
Specifically designed to recover strings that static tools miss. Emulates code to find strings decoded at runtime — defeats XOR, ROT, and other simple encoding schemes. Recovers the C2 URL from v4/v5 even though strings.exe shows nothing.
Detect-It-Easy (DiE)
Compiler / packer identifier — free
Identifies Go binaries by their runtime structure within seconds. Even stripped binaries are immediately identified as Go. Shows Go version used for compilation. Critical first step before attempting Ghidra analysis.
Ghidra
Reverse engineering suite — NSA/free
Full disassembly and decompilation. On unstripped binaries shows complete function names. On stripped Go binaries, used with GoReSym plugin to recover names. Can script XOR decoding to recover all encoded strings automatically.
GoReSym
Go symbol recovery — Mandiant
Recovers function names from stripped Go binaries by parsing the Go module metadata that the runtime requires. Defeats -ldflags="-s -w" stripping. Run as a standalone tool or as a Ghidra plugin.
strings.exe / strings2
String extractor — Sysinternals
Baseline first step. On unobfuscated builds extracts the C2 URL, commands, and paths immediately. On obfuscated builds, absence of expected strings confirms obfuscation is present — then escalate to FLOSS.

PEStudio walkthrough — what to look for

Download PEStudio from winitor.com. Drag the .exe onto it. The key panels:

Indicators
First panel to check. PEStudio flags suspicious characteristics automatically. For the payload you will see: process creation (exec.Command), file write (os.WriteFile), network access (net/http), registry modification (reg.exe via cmd). These are all flagged red.
Strings
On unobfuscated v3: immediately shows the C2 URL, HKCU\Software\Microsoft\Windows\CurrentVersion\Run, netstat -ano, tasklist /fo csv. On obfuscated v4/v5: no readable C2 strings. This confirms obfuscation is present.
Imports
Go binaries have a large number of imports because the standard library is compiled in. Look for CreateProcessW, RegSetValueExW, WSAStartup. These reveal the binary's capabilities regardless of obfuscation.
Entropy
XOR-encoded byte arrays show as slightly higher entropy sections compared to unobfuscated code. Entropy above 7.2 for a non-packed section is suspicious and worth investigating in Ghidra.
TLS / Signature
The binary is unsigned. No digital signature, no trusted publisher. PEStudio flags this immediately. Combined with the other indicators it confirms this is not legitimate software.

Defeating the obfuscation — analyst workflow

Each obfuscation technique implemented in Phase 04 has a corresponding analysis method. An experienced analyst defeats all three within about 20 minutes.

Defeating XOR string obfuscation (L07) with FLOSS

FLOSS (FLARE Obfuscated String Solver) emulates the binary's decoding routines and extracts the strings they produce at runtime. It recognises XOR decoding loops automatically. Download from github.com/mandiant/flare-floss.

Command Prompt — FLOSS against the obfuscated binaryshell
# Run FLOSS against the obfuscated binary
floss.exe v4_xor.exe

# FLOSS output will include the decoded strings:
# DECODED STRINGS (from emulated XOR routine):
http://192.168.1.10:5000/beacon
HKCU\Software\Microsoft\Windows\CurrentVersion\Run
ReconBeacon
netstat -ano
tasklist /fo csv
C:\temp\recon_combined.txt
XOR with a single byte key is defeated completely by FLOSS. FLOSS was specifically designed for this exact pattern. The decode() function in v4/v5 is a textbook XOR loop — FLOSS emulates it and prints every decoded string. The analyst has the C2 IP within seconds.

Defeating XOR in Ghidra manually

If FLOSS is not available, the same result is achievable in Ghidra in a few minutes:

1
Import the binary into Ghidra, let auto-analysis run. In the Symbol Tree, search for functions named main.decode (unstripped) or look for short functions with a loop containing XOR operations (stripped).
2
The decode function looks like: a loop, each iteration loads a byte, XORs with constant 0x5A, stores result. The constant 0x5A in the decompiler output is the XOR key.
3
In Ghidra's Script Manager, run a Python script to decode all byte arrays: find all byte[] data sections, XOR each byte with 0x5A, print readable output. This is a 10-line script.
4
Alternatively: open the binary in a hex editor, locate the encoded byte arrays (they are stored consecutively as global variables), copy them to a Python script and XOR decode manually.
Python — manual XOR decode of extracted bytespython
# Paste the encoded bytes from Ghidra here
encoded = [0x32,0x2e,0x2e,0x2a,0x60,0x75,0x75,0x6b,0x63,0x68,
           0x74,0x6b,0x6c,0x62,0x74,0x6b,0x74,0x6b,0x6a,0x60,
           0x6f,0x6a,0x6a,0x6a,0x75,0x38,0x3f,0x3b,0x39,0x35,0x34]

key = 0x5A
decoded = ''.join(chr(b ^ key) for b in encoded)
print(decoded)
# Output: http://192.168.1.10:5000/beacon

Defeating binary stripping (L09) with GoReSym

Download GoReSym from github.com/mandiant/GoReSym. It parses the Go module metadata embedded in the binary's .gopclntab section — this section is required by the Go runtime and cannot be removed with -ldflags.

Command Prompt — GoReSym against stripped binaryshell
GoReSym.exe v5_final.exe

# Output includes recovered function names:
# main.main
# main.sendToC2
# main.jitteredSleep
# main.decode
# Go version: go1.22.0
# Build path: C:\Users\sally\...\v5_jitter.go
GoReSym also recovers the build path. The output includes the full file system path where the binary was compiled — in this case exposing the operator's username and directory structure. This is a significant OPSEC failure that stripping alone does not address.

Network-layer obfuscation — what the payload does and how to detect it

Sleep jitter is the only network-layer obfuscation technique implemented in this project. Understanding exactly what it changes — and what it does not change — is important for both the attacker and the defender.

How to detect jittered beaconing

Jitter reduces the beacon score but does not eliminate it. Several detection approaches still work:

Method 1
Sustained capture + statistical analysis. Collect 30+ beacons (about 30 minutes). Calculate the mean and standard deviation of inter-arrival times. A legitimate user's browser makes requests with high variance. A jittered beacon has variance that clusters tightly around a mean — in this case around 60 seconds with a standard deviation of about 8 seconds. Export Wireshark stats: Statistics → Conversations, filter by the destination IP.
Method 2
Zeek with adjusted thresholds. The default Zeek beacon detection threshold catches periodicity scores above 0.7. Lowering to 0.5 catches ±20% jitter at the cost of more false positives. The trade-off is an operational decision.
Method 3
Protocol anomaly detection. Jitter does not change the HTTP headers. X-Host and X-Filename are still non-standard. A Suricata rule matching these specific headers fires regardless of timing: alert http any any -> any 5000 (msg:"ReconBeacon C2"; http.header; content:"X-Filename"; content:"X-Host"; sid:1000001;)
Method 4
Destination consistency. Every beacon goes to the same IP:port. Even with jitter, 30 POST requests to the same non-standard port from the same workstation over 30 minutes is anomalous. Network flow analysis flags this by destination frequency, not by timing.

What jitter does NOT obfuscate

Network indicatorAffected by jitter?Remaining detection
X-Host / X-Filename headersNo — unchangedSuricata rule on header names
Destination IP:portNo — same every beaconFlow frequency analysis

Plain HTTP (no TLS)No — POST body readableFull content inspection, DLP

What HTTPS would change. Upgrading the C2 server to TLS (adding a self-signed cert to Flask) would encrypt the POST body and headers — X-Host and X-Filename would no longer be visible in Wireshark. The URI (/beacon) would also be hidden. Destination IP and timing analysis would still work. This is the most significant remaining network detection gap in the current implementation.

VirusTotal — analysis and reporting

VirusTotal scans files and URLs against 70+ antivirus engines simultaneously and provides behaviour analysis from sandboxes. It is the fastest way to check whether a binary is detected by mainstream AV and to understand its capabilities without running it yourself.

How to submit a sample for analysis

Step 1
Go to virustotal.com. You do not need an account for basic analysis, but an account is required to view full behaviour reports and to submit files larger than 32MB. The payload binaries in this project are well under this limit.
Step 2
File tab: Click Choose file and select the .exe. VirusTotal will hash the file first — if the hash has been seen before, it returns cached results instantly without uploading. For the first submission of a new binary, the file is uploaded and scanned.
Step 3
URL tab: You can also submit the C2 URL (http://192.168.1.10:5000/beacon) to check if the IP or domain is flagged. A local lab IP (192.168.x.x) will not be flagged, but a real C2 domain would appear in threat intel databases if it has been reported before.
Warning
Files submitted to VirusTotal become public. Any premium user can download samples you submit. Never submit files containing credentials, personal data, or proprietary information. For lab binaries this is fine — for real incident response samples, submit only hashes or use a private scanning service.

Reading the results

Detection ratio
The headline number — e.g. "14/72" means 14 of 72 engines detected it. For a new Go binary with XOR obfuscation and symbol stripping, expect 0–5 detections on first submission. Without obfuscation (v3), expect 5–15 detections flagging the network behaviour and registry modification. A detection ratio above 30 means well-known malware with established signatures.
Detection names
Each engine reports its own name for what it found. Generic names like "Trojan.Generic" or "Malware.Heuristic" mean the engine flagged it by behaviour, not a specific signature. Named detections like "GoStealer.A" mean the engine has a specific signature for this family. For a custom-built educational payload, expect generic names if anything.
Details tab
Shows file metadata: SHA256, MD5, file size, PE header data, compilation timestamp, imports, sections. Check the imports list — for this payload you should see network, file system, process creation, and registry imports flagged.
Behaviour tab
Sandbox analysis from Cuckoo, Cape, and other automated sandboxes. This is the most useful section. Shows: processes created (cmd.exe children), files written (C:\temp\recon_combined.txt), registry modifications (HKCU Run key), network connections (POST to C2). This is behavioural analysis that XOR encoding and binary stripping do not defeat.
Relations tab
If the binary has been seen before, shows related files, URLs, and domains. For a fresh lab binary this will be empty. For known malware families this tab maps the infrastructure.

How to report malware to VirusTotal

If you analyse a binary and believe it is malicious but it is not detected by enough engines, you can submit it with context:

Method 1
Reanalyse button: If a file was previously scanned, click Reanalyse to trigger a fresh scan with current engine definitions. New signatures added since the last scan may now detect it.
Method 2
Submit to AV vendors directly: On the detection results page, for each engine that did NOT detect the file, there is a "Submit sample" link that sends it directly to that vendor's malware research team. This is how new signatures get added.
Method 3
NCSC / CISA reporting: For incidents involving actual systems, report to NCSC Ireland or CISA (US). These agencies coordinate threat intelligence sharing and issue advisories. VirusTotal analysis output is accepted as supporting evidence in incident reports.
Method 4
For this project specifically: Submitting the lab payloads to VirusTotal is a valid exercise — it shows what AV coverage looks like before and after obfuscation. The unobfuscated v3 binary versus the obfuscated v5 binary will likely have different detection ratios. Document both ratios in the project write-up.
