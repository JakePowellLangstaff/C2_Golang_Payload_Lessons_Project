How this payload would be detected

For every technique in this project, this page documents the detection method, the tool that surfaces it, and the signal strength. Understanding both sides is the point of the exercise.

Why blue team analysis matters here

Building an offensive tool without understanding how it is detected misses half the picture. This page maps every technique in the payload — from the cmd.exe child process to the registry Run key to the HTTP beacon interval — to the specific detection method a SOC analyst or EDR system would use to find it.

The detection coverage is deliberately layered. Most techniques are visible at multiple levels simultaneously: Sysmon catches the process event, Wireshark shows the network traffic, and Autoruns finds the persistence. A real defender uses all of these together, which is why the payload is noisy even at version 3.

Lab setup for this analysis: Sysmon with the SwiftOnSecurity config installed on the target VM. Wireshark running on the operator machine. Autoruns and Process Monitor from the Sysinternals Suite. All detection events described below are reproducible in the lab by following the instructions on the home page.

Network-based detection

Detection methodWhat it seesSeverity
Wireshark / packet captureX-Host and X-Filename headers visible in plaintext HTTP POST. Non-standard headers with "X-" prefix are an unusual pattern on port 5000 traffic.High
Zeek beacon detectionFixed 60-second inter-arrival time on outbound POST requests. Zeek's beacon detection script flags connections with a regularity score above threshold.High
Network flow logs (NetFlow / IPFIX)Repeated POST to same destination IP:port at regular intervals. Both the destination and the regularity are anomalous on a typical workstation.Medium
Firewall egress filteringOutbound traffic on TCP port 5000. Blocked by default in most enterprise environments. The payload would silently fail and retry on the next beacon cycle.Medium
IDS rule (Suricata / Snort)Can be written against the X-Host and X-Filename header combination, or against the /beacon URI path. Custom rules are straightforward once the pattern is known.Medium

Pyramid of Pain — what the payload bypasses and what it does not

The Pyramid of Pain (David J. Bianco, 2013) describes how difficult it is for an attacker to change each type of indicator once a defender detects and blocks it. Higher tiers cause more pain to the attacker. The table below maps each tier to this project's current payload.

TierExample in this projectBypassed by current payload?
TTPs Beaconing every 60s, Run key persistence, netstat/tasklist enumeration sequence No — core behaviour unchanged by any phase of this project
Tools The compiled .exe binary Partial — binary stripping + garble (Phase 04) hardens against static analysis signatures
Network Artifacts X-Host header, /beacon URI, port 5000, HTTP (not HTTPS) No — headers are hardcoded, channel is plaintext HTTP
Host Artifacts C:\temp\recon_combined.txt, "ReconBeacon" Run key value name No — static filenames and key names remain constant
IP Addresses 192.168.1.10:5000 hardcoded in binary Trivial — change the IP and recompile; hash changes too
Hash Values SHA-256 of the .exe Yes — any recompile produces a completely different hash

The practical conclusion is that this payload is easily detected at the Network Artifacts and Host Artifacts tiers — the tiers that are most useful to a working SOC analyst. Blocking a hash or an IP is trivial to evade. Detecting a registry Run key write or a beacon timing pattern is much harder for the attacker to change without altering core behaviour.
