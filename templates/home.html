What this project is

This site documents the development of a Go-based reconnaissance implant built for a final year cybersecurity project. The aim was to understand how offensive tooling is structured at the code level — not just conceptually — and to map each technique against real frameworks like MITRE ATT&CK and the Cyber Kill Chain.

Each lesson builds on the last. Version 1 collects basic host information and sends it to a Flask C2 server. Version 2 expands the data collection. Version 3 adds registry persistence and a timed beacon loop. The obfuscation phase covers string encoding, sleep jitter, and binary stripping. Everything runs in an isolated Windows VM — snapshotted before each test, Sysmon logging enabled, no internet access.

The Cyber Kill Chain

The Cyber Kill Chain (Lockheed Martin, 2011) is a seven-stage model of how a real intrusion unfolds from the attacker's side. Where MITRE ATT&CK catalogues individual techniques in isolation, the Kill Chain describes the overall campaign structure — each stage must succeed before the next can begin. It's one of the most widely used frameworks in both offensive security and threat intelligence.

This project works through the Kill Chain deliberately. Writing and compiling the Go payload is Weaponization. The Registry Run key is Installation. The Flask server receiving beacons every 60 seconds is Command and Control. Mapping the project to the Kill Chain is the clearest way to explain why each piece exists and what role it plays in the overall chain.

Delivery, Exploitation, and Actions on Objectives are out of scope — this is an isolated lab with no live targets. Stages 3, 4, and 7 are documented for completeness.

Installing Sysmon with the SwiftOnSecurity config

Sysmon is a Windows system driver that logs detailed events to the Windows Event Log — process creation, network connections, registry changes. On its own with default settings it logs too much to be useful. The SwiftOnSecurity config is a community-maintained XML file that filters out the noise and focuses on what actually matters for detection. Install both together.

Step 1
Download Sysmon from Microsoft Sysinternals: learn.microsoft.com/sysinternals/downloads/sysmon. Extract the zip. You'll have Sysmon.exe and Sysmon64.exe — use Sysmon64.exe on a 64-bit VM.
Step 2
Download the config from GitHub: github.com/SwiftOnSecurity/sysmon-config. Click Code → Download ZIP. The file you want inside is sysmonconfig-export.xml. Put it in the same folder as Sysmon64.exe.
Step 3
Open Command Prompt or PowerShell as Administrator (right-click the Start button → Terminal (Admin) or Command Prompt (Admin)) and run:
# Install Sysmon with the config — run as Administrator Sysmon64.exe -accepteula -i sysmonconfig-export.xml # Confirm it's running sc query sysmon64 # Expected: STATE : 4 RUNNING
Step 4
To view logs after running the payload: open Event Viewer (search for it in Start). Navigate to: Applications and Services Logs → Microsoft → Windows → Sysmon → Operational. You'll see EventID 1 when cmd.exe is spawned, EventID 3 for the outbound HTTP beacon, and EventID 13 for the registry Run key being written. This is what a real SOC analyst would see in a SIEM.
To update the config later: Sysmon64.exe -c sysmonconfig-export.xml  ·  To uninstall: Sysmon64.exe -u

Sysinternals Suite — three tools used in the lessons

Download the full Sysinternals Suite from learn.microsoft.com/sysinternals/downloads/sysinternals-suite and extract it to a folder on the target VM (e.g. C:\Tools\Sysinternals). No install required — every tool runs directly. Three are used throughout this project:

Autoruns
Run Autoruns.exe as Administrator. It lists everything configured to run at startup — registry Run keys, scheduled tasks, services, browser extensions. After running v3 of the payload, open the Logon tab. You'll see a ReconBeacon entry under HKCU\...\Run, highlighted yellow because the binary is unsigned and unknown to Windows. This is exactly how a defender or forensic analyst identifies persistence — it stands out immediately.
Process Monitor
Run Procmon.exe as Administrator before launching the payload. It captures every file, registry, and network event in real time. To filter to just your payload: click the filter icon, set Process Name → is → your_payload_name.exe. You'll see the exact sequence — directory creation, file write, registry key set, outbound connection — all timestamped. Useful for understanding what forensic artefacts remain on disk after the payload runs.
strings.exe
Run from the command line: strings.exe net_scan_v3.exe > strings_output.txt then open the text file. On an unobfuscated build you'll see the C2 URL (http://192.168.1.10:5000/beacon), the registry path, and command strings like netstat -ano all in plaintext. Run it again after the obfuscation lessons and compare. This before/after comparison is the clearest demonstration of why string obfuscation matters — save both outputs for your write-up.

Using Wireshark to capture and analyse beacon traffic

Wireshark captures all network traffic on a selected interface. For this project, run it on the operator machine (your PC), not the VM. You're watching the beacons arrive at your Flask C2 server on port 5000.

Step 1 — Install
Download from wireshark.org. During installation, agree to install Npcap when prompted — this is the packet capture driver Wireshark needs. Without it you won't see any traffic.
Step 2 — Pick the right interface
Open Wireshark. The interface list shows every network adapter on your machine. Select the one your VM traffic flows through. If using a VirtualBox Host-Only adapter it's labelled something like VirtualBox Host-Only Network. If using VMware NAT it might be your main Ethernet or Wi-Fi adapter. When you hover over each interface you'll see a live traffic graph — pick the one showing activity when the VM is running.
Step 3 — Filter and capture
Start the capture (blue shark fin button), then run the payload on the VM. In the display filter bar at the top type tcp.port == 5000 and press Enter. You'll see the TCP handshake (SYN, SYN-ACK, ACK) followed by the HTTP POST. Click the POST packet and expand Hypertext Transfer Protocol in the lower panel. You can read the X-Host and X-Filename headers in plaintext — this is why plain HTTP is a detection risk.
Step 4 — Analyse beacon timing
Change the filter to http.request.method == "POST" and let the capture run for 3–5 minutes. Look at the Time column — you'll see a POST arriving approximately every 60 seconds. This regularity is exactly what network detection tools like Zeek flag as a beacon. The timestamps are also useful evidence for the jitter lesson — screenshot this view, then screenshot again after jitter is added to show the intervals randomising.
Step 5 — Follow the TCP stream
Right-click any POST packet → Follow → TCP Stream. This reconstructs the full conversation and shows you the raw HTTP request (headers and file contents) and the Flask JSON response in one readable view. You can see the recon data being sent in cleartext. Save this as a screenshot for your project — it's a concrete visual of what exfiltration looks like on the wire before any encryption is added.
